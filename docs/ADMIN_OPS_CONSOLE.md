## 1. Назначение

Отдельное **административное** веб-приложение для операторов и разработчиков развёртывания: наглядный контроль состояния **фоновых процессов** (очереди, этапы пайплайна: транскрибация, диаризация, обобщение, индексация эмбеддингов), диагностика «зависло» (очередь Celery, остановленный воркер, отключённый этап в конфигурации), обзор активных и недавних разговоров **на уровне технических характеристик** (без доступа к содержимому чужих разговоров — см. §9) и при необходимости **управление процессами** в смысле пайплайна (повтор/отмена задач и т.п.), без управления контейнерами хоста (это требование снято — см. историю изменений).

Основное приложение (Web UI продукта) остаётся с **более широким кругом пользователей** и не обязано нести административные сценарии и права.

Админ-консоль **не** является копией основного приложения «плюс админка»: пользовательские сценарии создания контента и загрузки для обработки остаются в основном продукте (и при необходимости в CLI/API для автоматизации), см. §4.3.

## 2. Принятые архитектурные решения (зафиксировано)

| Решение | Формулировка |
|---------|----------------|
| Отдельное приложение | Да: отдельный фронтенд (и при необходимости минимальный BFF), не смешивать с основным Web UI. |
| Отдельный Admin API | Да: административные функции выносятся в **отдельный сервис** (`admin-api` или эквивалент), с отдельным ingress/образом при деплое. |
| Переиспользование с основным API | При необходимости часть операций выполняется **через существующий** пользовательский API или **общий код** (`send_task`, общие репозитории), чтобы не дублировать бизнес-логику пайплайна. |
| Не «второй продукт» | Оболочка над **теми же** сервисами инфраструктуры: те же БД, Redis, брокер очередей; ядро пайплайна не копируется «с нуля». |
| Идентичность и пользователи | Таблица **`users` единая** для продукта и админки: любой субъект входа — тот же пользователь сервиса. |
| Право администрирования | Отдельная таблица **членства администраторов** (название на усмотрение реализации, ниже — «таблица админов»): связь `user_id` → запись с **атрибутами административных ролей** (роли, метки, ограничения — по мере роста функций). **Саморегистрация админ-доступа невозможна**: запись в этой таблице создаётся только явно (bootstrap при миграции, последующее добавление оператором с достаточными правами или через прямой доступ к БД по процедуре). |
| Авторизация (единый стиль) | Токены и поток идентификации **единообразны** с основным приложением: тот же механизм выпуска и проверки **access JWT** (и при необходимости refresh), те же принципы, что в [AUTH_AND_IDENTITY.md](./AUTH_AND_IDENTITY.md). Отличие не в «типе логина», а в **авторизации**: эндпоинты Admin API допускают только запросы с валидным JWT **и** наличием строки в таблице админов для `user_id`. Обычный пользователь без такой строки не получает админ-функции даже при успешном входе в продукт. |
| Grafana / Flower | **Внешние** инструменты; из Ops-экрана — **ссылки** (новая вкладка). Встраивание через iframe **не** является обязательным и по умолчанию не требуется. |
| Язык интерфейса | **Русский** (один язык в MVP; i18n не требуется до отдельного решения). |
| Конфиденциальность | Администратор **не** имеет права **прослушивать**, просматривать текст расшифровки или иное **содержимое чужих разговоров** (разговоров других пользователей продукта). Допускаются только **общие технические характеристики** и управление процессами пайплайна — см. §9. |
| Управление контейнерами | Требование **аннулировано** (слишком сложно и рискованно для целевого MVP); управление инфраструктурой хоста — вне консоли, через привычные средства деплоя и внешние инструменты по ссылкам. |
| Не дублировать основной продукт | Админка — узкая Ops-поверхность (§1); **не** требуется загрузка аудио / создание разговоров для обработки из админ-консоли — это делается через **основное приложение** (или иные каналы продукта вне админки). |

## 3. Целевая аудитория и сценарии

1. **Разработка / отладка:** понять, на каком этапе застрял разговор, включены ли этапы в конфиге, есть ли воркеры и очередь.
2. **Эксплуатация (если применимо):** быстрый обзор «что сейчас обрабатывается», пользователь/размер/время, недавние ошибки.
3. **Технический аудит обработки (ограниченно):** по разговору — только **метаданные** версий транскриптов и статусы этапов (ревизия, `kind`, `status`, тайминги, обезличенные или короткие технические поля `meta`), **без** текста сегментов и без аудио.
4. **Управление процессами:** операции над **пайплайном** (перезапуск постановки задач ASR/diarization/summary/embeddings и т.д., расширяемый перечень), выполняемые через Admin API и общий код/внутренние механизмы — см. §4.1. Управление Docker/контейнерами **не** входит в требования.

## 4. Область функциональности (черновик MVP vs дальше)

Нумерация для последующей разбивки на задачи; приоритеты уточняются.

### 4.1 MVP (предварительно)

- **Вход:** тот же способ установления сессии, что и у основного приложения (единый стиль JWT/OAuth по §2); после входа админ-фронт обращается к **Admin API** с тем же Bearer-токеном; сервер отклоняет доступ без членства в таблице админов.
- **Обзор пайплайна:** таблица разговоров с фильтрами (статус ASR / диаризации / сводки сессии при включённой фиче, технические ошибки пайплайна), идентификаторы, пользователь (минимальный PII — см. §9), размер/длительность аудио **как числовые показатели** где доступно без дорогих запросов; **без** превью текста и без воспроизведения аудио.
- **Карточка / деталь разговора (админ):** только **техническая** сводка по транскриптам и связанным сущностям: ревизии, `kind`, `status`, тайминги, **очищенный** `meta` (без текстов расшифровки и без полей, эквивалентных содержимому разговора). Ссылку на основной продукт показывать только если это не обходит §9 (например без автологина «от имени владельца» в содержимое).
- **Управление пайплайном:** перечень операций **расширяемый**; типовые действия — повтор постановки ASR/diarization/summary/embeddings и т.п. (конкретизация в OpenAPI). Реализация через Admin API с вызовом **общего кода** или **внутреннего привилегированного** пути (без выдачи админу содержимого чужих разговоров); см. §10.
- **Инфраструктурный блок:** агрегированный статус зависимостей и очередей — см. §5.1.
- **Экран «Внешние инструменты»:** настраиваемые URL (Grafana, Flower, MinIO console и т.д.) из конфигурации деплоя.

### 4.2 Расширения поверх MVP (часть реализована в репозитории)

Ниже — изначальные **кандидаты** после baseline §4.1; по мере внедрения фиксируется фактическая картина (чеклисты спринтов 4–8, `openapi.yaml`).

**Уже есть в коде (кратко):**

- **Почти live-обновление:** HTTP-поллинг с jitter, экспоненциальным backoff и реже в фоновой вкладке браузера; лёгкий `GET /admin/api/v1/live-tick` без данных разговоров (спринт 7; решение — [adr/0002-admin-console-live-updates.md](./adr/0002-admin-console-live-updates.md)). Потоковый **SSE/WebSocket** для таблицы разговоров **не** используется.
- **Снимок эффективной конфигурации этапов:** `GET /admin/api/v1/pipeline-settings` без секретов (спринт 4).
- **Прогресс чанков ASR «k из N»:** колонки `transcripts.asr_chunk_completed` и `transcripts.asr_chunk_total`; в `GET /admin/api/v1/conversations` и в ревизиях карточки разговора; обновление из воркера ASR (параллельные слайсы и последовательное чанкование по времени) **без** помещения текста пользователя в `meta` (спринт 8).
- **Лента событий пайплайна:** таблица `pipeline_events`, `GET /admin/api/v1/pipeline-events` (пагинация, фильтры `conversation_id`, `event_type`), запись из воркеров ASR / диаризации / индексации эмбеддингов; во **admin-webui** — вкладка «События пайплайна» (спринт 8).

**В backlog / не сделано в первом релизе ленты:**

- Потоковый push (SSE/WS) при появлении новой строки в `pipeline_events` вместо только поллинга (опционально, спринт 8 C1).
- Политика **ретенции** и фоновая очистка `pipeline_events`.
- Расширение состава событий (например отдельные маркеры для LLM-сводки цепочки и др.) — по мере необходимости.
- **Спринт 9 (диагностика):** алерты инфраструктуры и очередей, сводка «требует внимания», таймлайн/диагноз в карточке разговора, коды ошибок в `pipeline_events`, сброс зависших ревизий — см. [ADMIN_OPS_SPRINT9_CHECKLIST.md](./ADMIN_OPS_SPRINT9_CHECKLIST.md).

### 4.3 Вне области (пока)

- **Мультитенантная** консоль для нескольких независимых деплоев из одного UI — **нет** (для v1).
- **Загрузка разговоров на обработку** и прочие пользовательские сценарии ввода контента из админки — **нет**: админка не повторяет основное приложение; загрузка и создание разговоров — через Web UI продукта (и при необходимости CLI/`POST /api/upload` с пользовательскими учётными данными вне админ-потока).
- Управление OAuth-клиентами провайдеров, массовое редактирование контента разговоров без явной операции в ТЗ.
- Полная замена Grafana/Flower собственными графиками.

## 5. Интеграция с существующим backend

**Канон:** отдельное приложение **Admin API** (отдельный процесс/образ), подключающееся к **той же** БД и Redis (и при необходимости вызывающее основной API или общие модули).

Проверка прав: общая таблица `users` + **JOIN** к таблице админов по `user_id`; общие секреты подписи JWT с основным API **или** явно согласованная пара issuer/secrets (избегать двух несовместимых форматов токена без необходимости).

**Единый JWT и два сервиса.** После успешного входа выдаётся **тот же** access JWT, что и для основного приложения: его подпись и срок действия проверяются **одинаково** и основным API, и Admin API (общий секрет/issuer и формат claims). Администратор — подмножество пользователей: его JWT **действителен** для обращения к основному API в обычных пользовательских маршрутах в пределах **своих** данных (как у любого `sub`).

Отдельный вопрос — не про «второй токен», а про **авторизацию на чужие разговоры**: пользовательские эндпоинты основного API обычно ограничены владельцем (`conversation.user_id == sub`). Чтобы **операционно** перезапустить пайплайн для разговора другого пользователя **без** выдачи админу содержимого (§9), при проектировании выбирают один из подходов:

1. **Расширение основного API:** те же JWT; dependency проверяет членство в таблице админов и для явно помеченных маршрутов **разрешает** действие над любым `conversation_id` с аудитом (ответ по-прежнему без содержимого, если это админ-контракт).
2. **Только Admin API:** мутации выполняются только здесь через **общий код** (постановка Celery и т.д.) после проверки JWT + админ-членства; основной API для этого не вызывается.

«Привилегированный слой» в требованиях означает **не отдельный тип Bearer-токена**, а место в коде, где безопасно выполняется операция над ресурсом **не текущего** владельца при сохранении §9.

### 5.1 Инфраструктурный блок — минимальный набор сигналов

- Доступность **PostgreSQL** (чтение или простой запрос).
- Доступность **Redis** (брокер Celery).
- **Ответ основного API** (`/health` или лёгкий ping) — чтобы отличить «админ жив, продукт упал».
- По **очередям Celery**: хотя бы сводка по **именованным очередям** из [`server/workers/celery_app.py`](../server/workers/celery_app.py): `asr_fast`, `asr_final`, `asr`, `diarization`, `llm` (включая embeddings), `cleanup`.

Точный вид метрик (глубина очереди vs только «есть ли worker», период опроса, кэширование) — решение при реализации.

Источники данных для обзора разговоров: модели `Conversation`, `Transcript`, `RecordingSessionSummary`, `Embedding`; при необходимости — Celery inspect и ключи Redis. Поля с текстом расшифровки и скрытые артефакты аудио в ответах Admin API **не включаются** (§9); `Transcript.meta` допускается только после **очистки** от текстовых фрагментов содержимого, если они там могут появиться.

## 6. Модель административной аутентификации и членства

- **Единый стиль авторизации** с основным приложением: см. §2 («Авторизация (единый стиль)»).
- **Таблица админов:** одна или несколько записей на пользователя; поля для **административных ролей и атрибутов** (расширяемо). Отсутствие записи означает отказ во всех admin-маршрутах.
- **Первый администратор:** создаётся при **инициализации схемы БД**: строка в `users` (или использование существующего механизма создания пользователя) **плюс** строка в таблице админов; начальный пароль/идентичность — по правилам bootstrap (env + миграция). **Получить админ-доступ через самостоятельную регистрацию в продукте нельзя.**
- **Коррекция членства и данных при необходимости:** допускается **прямое изменение в БД** (организационная процедура + документация), без обязательного полного UI управления ролями в MVP.
- Требования к паролям продуктового пользователя, блокировке входа, аудиту входов и **всех административных действий** — зафиксировать минимально для MVP.

## 7. Развёртывание и конфигурация

- Отдельный образ **Admin API** + статическая сборка **админ-фронта** или отдача через reverse proxy под префиксом `/admin` или отдельным хостом `admin.example.com`.
- Переменные окружения: те же параметры, что нужны для проверки JWT (или общий модуль конфигурации с основным API); URL основного API при необходимости; строки БД/Redis; URL Grafana/Flower и др.; режим поллинга.
- **Почти live-обновление консоли (спринт 7):** по умолчанию admin-webui использует HTTP-поллинг с джиттером, backoff при ошибках и реже опрашивает данные, когда вкладка браузера в фоне (см. `docs/adr/0002-admin-console-live-updates.md`). Если данные «застыли», проверьте сеть, срок JWT и кнопку «Обновить сейчас»; при подозрении на зависший прокси включите на сервере лог уровня DEBUG для маршрута `GET /admin/api/v1/live-tick` (в лог пишется только `user_id` вызывающего админа).
- Локальная разработка: описать в [docker/README.md](../docker/README.md) после проектирования сервисов.

## 8. Нефункциональные требования (черновик)

- **Безопасность:** админ-маршруты недоступны анонимно и недоступны без строки в таблице админов; соблюдение §9 на уровне API (не отдавать текст транскрипта и не отдавать потоки/URL аудио чужих разговоров).
- **Производительность:** списки с пагинацией; тяжёлые запросы (S3 HEAD для каждой строки) — опционально или кэш.
- **Наблюдаемость:** логирование админ-действий (минимум: вход, просмотр сводок по разговорам, операции управления пайплайном).

## 9. Данные и конфиденциальность

**Жёсткое правило:** администратор **не** получает доступ к **содержимому чужих разговоров**. Под запретом: **прослушивание аудио**, отображение **текста расшифровки** (сегменты, markdown/json транскрипта), экспорт содержимого, любые действия, эквивалентные чтению разговора от имени владельца.

**Разрешено** в отношении разговоров **любых** пользователей продукта (наблюдение и эксплуатация):

- идентификаторы (`conversation_id`, `user_id`, при необходимости краткий стабильный ярлык без обязательного email);
- **технические** статусы и этапы пайплайна (ASR, diarization, summary, embeddings и т.д.);
- **агрегированные** показатели: длительность, размер файла, число ревизий, признаки чанков из `meta`, где они **не** воспроизводят текст;
- короткие **сообщения об ошибках** пайплайна (без включения пользовательского текста из транскрипта);
- управление **процессами** (повтор/отмена фоновых задач) в объёме, согласованном в OpenAPI, **без** необходимости показывать админу текст расшифровки.

Для **собственных** разговоров администратора-как-пользователя действует обычная модель продукта (основное приложение); админ-консоль **не** обязана дублировать пользовательский просмотр содержимого и не должна использоваться как обход запрета на чужие данные.

---

## 10. Для проектирования реализации (не блокируют v1.0)

На этапе спецификации API и кода остаётся выбрать технический вариант — он **не меняет** согласованный объём админки и правила §9:

1. **Где реализовать мутации пайплайна по чужим разговорам:** подход **(1)** или **(2)** из §5 (расширение основного API для админов vs мутации только в Admin API через общий код) — решение по дублированию кода, поверхности атаки и сопровождению; во всех вариантах — **один JWT**, без отдельного «админ-токена» и без impersonation владельца для доступа к содержимому (§9).

Прочие детали (минимальная политика паролей, полнота аудит-лога, точный список мутаций в первом релизе) выносятся в задачи реализации и при необходимости в `openapi.yaml`, без возврата к пересмотру v1.0, пока не потребуется явное изменение продукта.

**Закрыты на уровне требований v1.0:** отдельный Admin API; единая `users` + таблица админов; единый стиль JWT/OAuth с разграничением по членству; без саморегистрации админов; расширяемый перечень операций управления **пайплайном**; без мультитенантности; русский UI. **Управление контейнерами** — требование снято. **Доступ к содержимому чужих разговоров** — запрещён (§9). **Загрузка разговоров на обработку из админки** — **не требуется**: админка не дублирует основное приложение (§4.3, §2).

## 11. Связь с задачами на реализацию

Исходный план ниже **в основном закрыт** в репозитории (сервисы, миграции, маршруты, `admin-webui`, Docker — см. чеклисты спринтов 2–8). Дальнейшие задачи — расширения из §4.2 (backlog) и сопровождение.

Ожидаемые направления работ (исторически):

1. Миграции БД: таблица **членства администраторов** (`user_id` FK → `users`, поля ролей/атрибутов) + bootstrap первого админа (строка в `users` при необходимости + строка в таблице админов).
2. Согласование **проверки JWT** между основным API и Admin API (общий секрет/модуль).
3. Новый сервис **Admin API**: зависимость «JWT + членство в таблице админов», обзор пайплайна (только технические поля по §9), инфраструктурный блок, операции управления пайплайном по выбранной схеме §10 п. 1 / §5.
4. Интеграция с **общими модулями** постановки задач / репозиториями — без дублирования правил пайплайна и без обхода §9.
5. Фронтенд админ-приложения (русский UI): единый поток входа с основным приложением, затем работа только с Admin API; UI без проигрывателя аудио и без текста чужих транскриптов.
6. Docker Compose / документация деплоя сервисов **admin-api** и админ-фронта.

**История изменений документа:**

- Начальная версия — отдельное приложение, отдельные учётные записи, Grafana/Flower как внешние ссылки.
- Уточнение — отдельный Admin API; первый админ из инициализации БД; управление не только просмотр; без мультитенантности; русский UI; инфраструктурный блок и очереди Celery.
- Уточнение — **единая таблица `users`**, отдельная таблица админов с ролями/атрибутами; **единый стиль авторизации** с основным приложением (JWT/OAuth), админ-доступ только при явном членстве, без саморегистрации; расширяемое управление **процессами пайплайна**; вариант с управлением **контейнерами** обсуждался и **снят** в следующей редакции.
- Уточнение — требование **управления контейнерами** **аннулировано**; зафиксирован **запрет** на прослушивание и просмотр **содержимого чужих разговоров**; разрешены только **общие технические характеристики** и **управление процессами пайплайна** (§9, §2).
- Уточнение — админка **не** является повтором основного приложения; **загрузка разговоров на обработку из админки не требуется** (§4.3, §2); открытый вопрос в §10 по загрузке **закрыт**.
- Уточнение — **единый JWT** принимается и основным, и Admin API; «привилегированный слой» в тексте требований относится к **реализации операций над чужими разговорами**, а не к отдельному типу токена (§5, §10).
- **Версия 1.0 (baseline)** — статус документа переведён на зафиксированные требования; §10 переименован в «для проектирования реализации».
- **2026-05-14** — §4.2: зафиксирована фактическая реализация (поллинг v2, `live-tick`, `pipeline-settings`, прогресс ASR по колонкам, `pipeline_events` + эндпоинт и вкладка UI); отдельно указан backlog (SSE-пуш ленты, ретенция, расширение типов событий).
