# Ops-консоль: спринт 3 — чеклист задач и порядок PR

**Связь:** [ADMIN_OPS_CONSOLE.md](./ADMIN_OPS_CONSOLE.md) (§4.1 расширение операций, §6–§8 наблюдаемость и аудит, §4.2 поллинг).  
**База:** спринт 2 (список/деталь, мутации retranscribe + reindex-embedding, admin-webui, инфраструктура).

Ниже — разложение в чеклист с **рекомендуемым порядком мержей** (от меньшей зависимости к большей). Отмечайте пункты по мере выполнения (`[x]`).

**Статус реализации (код):** G1–I3 выполнены в репозитории; Definition of Done — закрыт (см. историю 2026-05-14).

## Порядок PR (рекомендуемый)

1. **PR-1** — Epic G (аудит в БД): миграция + модель + запись событий; не меняет публичный контракт списка разговоров.
2. **PR-2** — Epic G (продолжение): `GET /admin/api/v1/audit-events` + запись просмотра карточки (`conversation_view`); тесты 401/403.
3. **PR-3** — Epic H (мутация **rediarize**): общий паттерн с §9, Celery `run_diarization`, аудит.
4. **PR-4** — Epic I (admin-webui): вкладка аудита, поллинг с backoff для «Разговоры» и «Инфраструктура», кнопки мутаций на карточке.

---

## PR-1: Таблица аудита и запись (Epic G)

- [x] **G1** Alembic: таблица `admin_audit_events` (`id`, `admin_user_id` FK → `users`, `action`, `conversation_id` nullable UUID, `detail` text nullable, `created_at` timestamptz); индекс `ix_admin_audit_events_created_at` по `created_at`.
- [x] **G2** ORM-модель `AdminAuditEvent` в `app/models.py` (кратко в шапке файла).
- [x] **G3** Функция `record_admin_audit_event` в `admin_api/audit.py`: структурированный лог + вставка строки в **отдельной** транзакции (`session_scope`), обрезка `detail` по длине.
- [x] **G4** Заменить вызовы `log_admin_pipeline_action` на `record_admin_audit_event` в мутациях пайплайна (retranscribe, reindex-embedding).

---

## PR-2: Просмотр аудита и событие «открыл карточку» (Epic G)

- [x] **G5** Репозиторий/запрос: пагинация `limit`/`offset`, опциональные фильтры `action`, `conversation_id`, `admin_user_id` (все — только для пользователя с `admin_memberships`).
- [x] **G6** `GET /admin/api/v1/audit-events`: Pydantic `extra = "forbid"`; в ответе **нет** содержимого разговоров (только id и технические поля события).
- [x] **G7** После успешного `GET /admin/api/v1/conversations/{id}` — запись `conversation_view` (не писать при 404).
- [x] **G8** Тесты: `GET /audit-events` без Bearer → 401; с override админа → 200 и форма `items`/`total`/`limit`/`offset`.

---

## PR-3: Повтор диаризации из админки (Epic H)

- [x] **H1** Вспомогательная проверка «диаризация уже running» по аналогии с продуктовым `POST .../diarize` (`kind == asr_diarized`, `status == running`).
- [x] **H2** `POST /admin/api/v1/conversations/{id}/actions/rediarize` → 202, очередь `diarization`, `args` как у воркера; 400 если `diarization.enabled=false`; 409 при running.
- [x] **H3** Аудит `action=rediarize`; тест 401 без токена.

---

## PR-4: Admin UI — поллинг и операции (Epic I)

- [x] **I1** Вкладка «Аудит»: загрузка `GET /audit-events?limit=50` (ручная кнопка «Обновить» + опционально тот же backoff-поллинг, пока вкладка активна).
- [x] **I2** На вкладках «Разговоры» и «Инфраструктура»: автообновление с **экспоненциальным backoff** (старт ~5 с, потолок ~60 с), сброс backoff при смене вкладки или после ручного «Загрузить».
- [x] **I3** Вкладка «Карточка»: кнопки «Повтор ASR», «Переиндексировать embedding», «Повтор диаризации» (POST на соответствующие пути), отображение краткого результата/ошибки.

---

## Финальная приёмка спринта 3 (Definition of Done)

- [x] Миграции применяются на чистой БД; админ-API и unit-тесты зелёные. *(в т.ч. через сервис `tests` в compose.)*
- [x] В БД появляются строки при мутациях и при открытии карточки разговора.
- [x] Обычный JWT без `admin_memberships` не получает `/audit-events` и новый `rediarize`.
- [x] Ручная проверка: поллинг не забивает брокер/БД (потолок интервала соблюдён). *(см. ADR спринта 7 для уточнённого поллинга v2.)*
