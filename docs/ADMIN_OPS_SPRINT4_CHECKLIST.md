## Порядок PR (рекомендуемый)

1. **PR-1** — Epic J (аудит использования админ-API): запись при `GET /me` как сигнал «сессия консоли»; тесты 401.
2. **PR-2** — Epic K (read-only **эффективная конфигурация** пайплайна): `GET /admin/api/v1/pipeline-settings` без секретов; тесты формы и 401.
3. **PR-3** — Epic L (инфраструктура §5.1): при наличии Redis-брокера — **оценка глубины** именованных очередей (`LLEN` по имени очереди) в том же TTL-кэше, что и Celery inspect; не ломать ответ при ошибке.
4. **PR-4** — Epic M (admin-webui): вызов `GET /me` при смене сохранённого токена (аудит сессии); вкладка «Пайплайн» с JSON снимка; при необходимости краткая подсказка в UI.

## PR-1: Аудит «сессии» консоли (Epic J)

- [x] **J1** В обработчике `GET /admin/api/v1/me` после успешной авторизации вызвать `record_admin_audit_event` с `action=admin_console_session` (без PII в `detail`; при необходимости — только техническая метка вроде `source=api_me`).
- [x] **J2** Ошибки записи аудита не должны ломать ответ `200` `/me` (как для `conversation_view`).
- [x] **J3** Тест: без Bearer → 401; с override админа → 200 и в коде пути вызывается запись аудита (mock `record_admin_audit_event`).

## PR-2: Снимок конфигурации пайплайна (Epic K)

- [x] **K1** Pydantic-модели ответа `extra = "forbid"`: ASR (default_provider, recognition_model, провайдеры: имя, enabled, model без путей к секретам на диске — достаточно флагов и публичных имён), диаризация (enabled, default_provider, turn_level_retranscription, провайдеры: имя, enabled, impl/model/device на усмотрение без токенов), LLM (session_summary_enabled, лимиты символов, провайдеры: enabled, **без** api_key), embeddings (enabled, provider, model, **без** ключей), limits (chunk_ms_min/max, realtime modes, max_window_ms, autoprolong).
- [x] **K2** `GET /admin/api/v1/pipeline-settings`: только `require_admin_principal`; данные из уже загруженного `app_config` (эффективный снимок после env overrides).
- [x] **K3** Явно **не** включать: OAuth client_secret, S3 secret_key, database URL, Redis URL, LLM/embedding API keys, полные пути к HF-токенам в `detail` (если выводите `hf_token_env` — только **имя** переменной окружения, не значение).
- [x] **K4** Тест: 401 без токена; 200 с админом и проверка отсутствия запрещённых ключей в JSON (`api_key`, `secret`, `openai_api_key` и т.д.).

## PR-3: Глубина очередей Celery в инфраструктуре (Epic L)

- [x] **L1** Расширить элемент `celery_queues`: поле `queue_depth: int | null` (число сообщений в списке Redis для имени очереди, или `null` если брокер недоступен / формат неизвестен).
- [x] **L2** Использовать тот же кэш (~10 с), что и для `inspect.active_queues`, чтобы не удваивать нагрузку на поллинг UI.
- [x] **L3** Unit-тест или мок: ответ сериализуется, поле присутствует; при ошибке Redis — `null` без 500.

## PR-4: Admin UI (Epic M)

- [x] **M1** При старте с токеном в `localStorage` и по кнопке «Сохранить токен локально» вызвать `GET /admin/api/v1/me`, чтобы зафиксировать `admin_console_session` в аудите (не вызывать `/me` на каждое изменение символов в поле).
- [x] **M2** Вкладка «Пайплайн»: загрузка `GET /admin/api/v1/pipeline-settings`, ручное «Обновить» + опционально backoff-поллинг только пока вкладка активна (как «Аудит»).
- [x] **M3** Краткий текст: снимок для отличия «выключено в конфиге» от «в очереди».
