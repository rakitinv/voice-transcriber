# Ops-консоль: спринт 6 — чеклист задач и порядок PR

**Связь:** [ADMIN_OPS_CONSOLE.md](./ADMIN_OPS_CONSOLE.md) (§6 членство и политики, §8 наблюдаемость и аудит), [AUTH_AND_IDENTITY.md](./AUTH_AND_IDENTITY.md), [ADMIN_OPS_ROADMAP.md](./ADMIN_OPS_ROADMAP.md).  
**База:** спринт 5 (гигиена, список §4.1, OpenAPI, мутации).

**Статус реализации (код):** **закрыт** — PR-1–PR-4 и Definition of Done выполнены (2026-05-14; позже: ретенция `auth_signin_events` из конфига/env, `.env` / compose для аудита и admin OAuth).

---

## Порядок PR (рекомендуемый)

1. **PR-1** — Epic R (§6 минимум): политика паролей / блокировки входа в **продуктовом** API — только то, что согласовано с ТЗ; без расширения админки «вторым продуктом».
2. **PR-2** — Epic S (аудит входа в продукт): события успешного/неуспешного логина (формат, хранение, ретенция) — по согласованию с безопасностью и GDPR.
3. **PR-3** — Epic T (аудит Admin API read): при необходимости политики — запись в `admin_audit_events` для чтений (список разговоров, инфраструктура, `pipeline-settings`, `external-tools`) с дедупликацией или семплированием, чтобы не раздувать таблицу.
4. **PR-4** — Epic U (**сделано**): OAuth во `admin-webui` + allowlist `VT_ADMIN_WEBUI_*` на API; тот же callback провайдера, что у Web UI.

## PR-3: Расширение аудита админ-чтений (Epic T)

**Уточнить при старте спринта:** нужен ли аудит каждого GET или выборочно (например только `pipeline-settings` и список); лимиты на частоту записи.

## PR-4: OAuth во admin-webui (Epic U)

**Уточнить при старте спринта:** тот же client_id, отдельный redirect URI для `admin.example.com` или общий — с DevOps.

Решение: отдельный redirect URI у провайдера **не** нужен; только allowlist origin админ-SPA на API (`VT_ADMIN_WEBUI_ORIGIN` / `VT_ADMIN_WEBUI_ORIGINS`).

- [x] **U1** Конфиг `VITE_*` для OAuth (authority, client id, redirect).
- [x] **U2** Обмен code → tokens совместимый с основным API; сохранение access JWT как сейчас.
- [x] **U3** Fallback: ручная вставка JWT остаётся для отладки (если команда согласна).
