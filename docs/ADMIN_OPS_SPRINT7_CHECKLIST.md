# Ops-консоль: спринт 7 — чеклист задач и порядок PR

**Связь:** [ADMIN_OPS_CONSOLE.md](./ADMIN_OPS_CONSOLE.md) (§4.2 почти live-обновление), [ADMIN_OPS_ROADMAP.md](./ADMIN_OPS_ROADMAP.md).  
**База:** спринт 6 (политики, аудит, опционально OAuth в admin-webui).

**Статус реализации (код):** спринт 7 — см. ADR `docs/adr/0002-admin-console-live-updates.md`, `GET /admin/api/v1/live-tick`, `useAdminTabAutoRefresh` в admin-webui.

---

## Порядок PR (рекомендуемый)

1. **PR-1** — Epic V (выбор механизма): ADR или короткое решение в репозитории — SSE vs WebSocket vs улучшенный поллинг (единый клиентский хук, backoff, пауза в фоне).
2. **PR-2** — Epic W (Admin API): если выбраны SSE/WS — эндпоинт(ы) потока **без** содержимого чужих разговоров (§9); только технические события или «тик» для обновления счётчиков.
3. **PR-3** — Epic X (admin-webui): подписка/поллинг v2; отписка при смене вкладки и unmount; лимиты частоты согласованы с §8.
4. **PR-4** — Epic Y (наблюдаемость): метрики или логи «подключений» к потоку; защита от утечки соединений.

## PR-1: Архитектурное решение (Epic V)

**Уточнить при старте спринта:** нужен ли серверный push или достаточно централизованного поллинга с jitter; совместимость с reverse proxy (таймауты, буферизация SSE).

- [x] **V1** Документ решения (1–2 страницы): мотивация, риски, откат на текущий поллинг.
- [x] **V2** Согласование с DevOps (ingress timeouts, sticky sessions если WS).

## PR-2: Реализация на стороне Admin API (Epic W)

**Уточнить при старте спринта:** объём данных в потоке (только «версия снапшота» vs мини-дельты).

- [ ] **W1** Если SSE: `text/event-stream`, корректные заголовки, закрытие по idle. *(не реализовано: по ADR выбран улучшенный HTTP-поллинг и `GET /admin/api/v1/live-tick` без SSE.)*
- [x] **W2** Авторизация: тот же Bearer + `admin_memberships`; отключение при истечении JWT.
- [x] **W3** Тесты: 401 без токена; 403 без админ-строки; отсутствие §9-нарушений в payload.
