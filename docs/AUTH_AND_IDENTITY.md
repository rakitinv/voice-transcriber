# Аутентификация, идентичность и сессия (Web UI, расширение, единый backend)

**Статус:** канон требований (живой документ).  
**Связанные документы:** [TECHNICAL_SPECIFICATION.md](./TECHNICAL_SPECIFICATION.md) (§2, §3.3–§3.4, §15), [BROWSER_EXTENSION_UI.md](./BROWSER_EXTENSION_UI.md), [ROADMAP.md](./ROADMAP.md).  
**Код (ориентиры):** `server/app/api/auth.py`, `server/core/security.py`, `browser-extension/src/auth/oauth.ts`.

---

## 1. Цели и границы

1. **Единый backend** для всех установок данного расширения: OAuth-клиенты провайдеров привязаны к одному продукту/развёртыванию; расширение не реализует собственный OAuth без сервера.
2. **Расширение не хранит и не обрабатывает** долгоживущие токены провайдера (Google, Yandex и др.). Оно работает только с **токенами сервиса** (короткоживущий access JWT и при внедрении — **refresh-секрет сервиса**, см. §4).
3. **Провайдер OAuth** используется **только для установления идентичности** (однозначный субъект: `sub` / issuer+subject, при необходимости отображаемый email из userinfo или id_token). **Сохранять `refresh_token` провайдера на backend для текущих сценариев не требуется.**
4. **«Тихий» UX входа** достигается комбинацией:
   - попытки **без UI у провайдера**, если у пользователя уже есть сессия у провайдера в этом браузере (см. §3);
   - после успешной идентификации — **долгой сессией сервиса** без повторного OAuth (см. §4).
5. **Слияние учётных записей** (один пользователь продукта — несколько способов входа): **только через Web UI**; расширение лишь выбирает провайдера для **однозначной** идентификации по уже смоделированным связям на сервере.

---

## 2. Архитектура потока (расширение)

```
[Расширение]  --launchWebAuthFlow-->  [Провайдер OAuth]
       |                                      |
       |<-- redirect с code на chromiumapp --|
       |
       +-- POST /api/auth/{provider}/extension/finalize?code=...
                 |
          [Backend: обмен code, извлечение identity, выдача токенов сервиса]
```

- Секреты провайдера (`client_secret` при необходимости) и любые **серверные** refresh-секреты остаются **только на backend**.
- Проверка «залогинен ли пользователь в API»: **`GET /api/auth/me`** с `Authorization: Bearer <access JWT>`; продление сессии: **`POST /api/auth/refresh`** (тело и ответ — в **`openapi.yaml`**, реализация C7.2).

---

## 3. Гибридный UX в расширении (провайдер)

1. **Первый шаг — неинтерактивная попытка:** `chrome.identity.launchWebAuthFlow` с **`interactive: false`**, в URL авторизации провайдера — параметры в духе **«без UI, если сессия уже есть»** (например Google `prompt=none`; для Яндекса — согласованный с документацией провайдера аналог).
2. **При отсутствии кода / ошибке провайдера** — **вторая попытка** с **`interactive: true`** и обычным authorize (в т.ч. выбор аккаунта, если продуктово требуется).
3. Ожидаемо, что silent-шаг **часто завершается неуспехом** (нет сессии у провайдера, политика браузера): клиент **не показывает** это как сбой, а **сразу** переходит к интерактивному шагу без лишнего шума в UI.

Детали отображения в popup — в [BROWSER_EXTENSION_UI.md](./BROWSER_EXTENSION_UI.md).

## 4. Долгая сессия сервиса (без повторного OAuth)

- **Короткий** access JWT (как сейчас по смыслу в `server/core/security.py`) остаётся основным средством авторизации REST/WS.
- Для редкого показа провайдера вводится **отдельный механизм продления только на стороне сервиса**, например:
  - **refresh-токен** сервиса (хранение, ротация, отзыв по устройству/сессии), выдаваемый после успешного OAuth finalize; клиент расширения хранит его в **`chrome.storage`** с пониманием модели угроз; **или**
  - сессионная модель для Web UI (**httpOnly cookie**) там, где применимо; расширение при этом остаётся на Bearer + refresh в storage, если не вводится отдельный device-flow.
- Контракт HTTP зафиксирован в **`openapi.yaml`** (`ExtensionFinalizeResponse`, `AuthRefreshRequest` / `AuthRefreshResponse` и пути `extension/*`, `/api/auth/refresh`).

**Не путать** с §3: «тихий» шаг у провайдера — опциональное улучшение UX; **основной** механизм долгой работы без провайдера — **refresh (или аналог) сервиса**.

## 5. Модель данных идентичности и слияние (Web UI)

- Пользователь продукта — сущность **`users`** (или эквивалент); связь с провайдером — отдельная таблица вида **`user_oauth_accounts`** / **`auth_identities`**: `user_id`, `provider`, **`provider_subject`** (стабильный `sub`), метаданные (email на момент входа), временные метки.
- Поиск пользователя при входе: по паре **`(provider, provider_subject)`**; email **не** считается первичным ключом идентичности без дополнительных мер.
- **Слияние аккаунтов:** поток только в **Web UI** (подтверждение владения вторым провайдером, перенос связей на один `user_id`, разрешение конфликтов данных). После слияния вход через любой привязанный провайдер ведёт к одному пользователю — при условии, что backend при логине разрешает несколько идентичностей на одного пользователя (см. реализацию).

## 6. Безопасность и качество реализации

1. **PKCE** для публичного клиента расширения (рекомендуется к обязательному требованию для production): `code_challenge` / `code_verifier`, проверка на finalize.
2. **State** или эквивалентная привязка запроса finalize к исходной сессии авторизации, защита от подмены и повторного использования кода — по рекомендациям OAuth 2.0 для native/public clients.
3. **Redirect URI** для расширения — только `https://<extension-id>.chromiumapp.org/...` (уже проверяется на backend в зачаточной форме).
4. **Client_id** провайдера: при едином backend допускается конфигурация в репозитории; предпочтительно **не дублировать** логику построения URL authorize в расширении и на сервере — использовать ответ API вида **`GET /api/auth/{provider}/extension/start?redirect_uri=...`** (уже частично предусмотрено в коде).

---

## 7. Состояние реализации на момент фиксации

- Backend: обмен `code` и вызовы userinfo у провайдеров в **`auth.py`** на момент документа могут быть **заглушками**; требования настоящего документа задают **целевое** поведение.
- Расширение: уже использует `launchWebAuthFlow` + `POST .../extension/finalize`; гибрид silent→interactive и refresh сервиса — **к внедрению** по дорожной карте.

## 8. Политика учётных данных (MVP §6 продукта, 2026-05)

На текущем этапе **основной вход в продукт** — OAuth провайдеров (Google, Яндекс) и выдача **короткого access JWT** плюс **сервисного refresh** (`POST /api/auth/refresh`). Отдельного потока **регистрации по email/паролю** в API нет; утилиты хеширования пароля в коде зарезервированы на будущее.

1. **Пароли (если появятся в API):** минимальная длина **12** символов; без ведущих/завершающих пробелов; не короче и не длиннее разумных пределов реализации; классы символов — минимум **три из четырёх**: строчные, прописные, цифры, спецсимволы (см. `server/app/services/password_policy.py`).
2. **Блокировка перебора (lockout):** по **IP клиента** (с учётом `X-Forwarded-For`, первый hop) в **Redis** ведутся счётчики неуспешных попыток для **`POST /api/auth/refresh`** (невалидный refresh) и для заголовка **`X-VT-Api-Key`** (неверный ключ). Пороги и окна задаются в `configs/server.yaml` → `auth.lockout` и переменными окружения `VT_AUTH_LOCKOUT_ENABLED`, `VT_REDIS_URL`. При превышении порога на окно ответ **`429 Too Many Requests`** с заголовком `Retry-After`. Успешный refresh сбрасывает счётчик неуспехов для данного IP (смягчение для общих NAT).
3. **Ops admin-webui:** вход через те же `GET /api/auth/google|yandex` с `client=admin` и `next=<origin SPA>`; редирект с токенами в fragment на **origin из allowlist** (`VT_ADMIN_WEBUI_ORIGIN` / `VT_ADMIN_WEBUI_ORIGINS`). У **Google/Яндекс** в консоли OAuth по-прежнему только **callback API** (`…/api/auth/*/callback`), без отдельного redirect URI на хост админки.
4. **Аудит входа:** таблица **`auth_signin_events`** (миграция Alembic): исход `success` / `failure`, канал (`oauth_web`, `oauth_extension`, `refresh`, `api_key`, `oauth_link`), код причины, опционально `provider`, опционально `user_id`, опционально **отпечаток клиента** (SHA-256 от **`VT_AUTH_AUDIT_SALT`** + IP — **не** хранится сырое значение IP в явном виде). **Срок хранения** задаётся без правки кода: `configs/server.yaml` → `auth.login_audit.retention_days` (по умолчанию 90) или переменная окружения **`VT_AUTH_SIGNIN_EVENTS_RETENTION_DAYS`**. Значение **`0`** отключает автоматическое удаление. Очистка выполняется задачей Celery **`workers.tasks.cleanup.old_auth_signin_events`** (ставится в очередь вместе с `workers.tasks.cleanup.schedule_cleanup` → `cleanup_expired_conversations`); при отсутствии Beat задачу можно вызывать вручную или повесить на внешний cron. Тем же сроком (`retention_days`) по расписанию удаляются и строки **`pipeline_events`** (лента пайплайна Ops-консоли) — задача **`workers.tasks.cleanup.old_pipeline_events`**.
