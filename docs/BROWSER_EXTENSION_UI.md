# Расширение браузера (Chromium): UI и поведение — соглашения (2026-04)

Настоящий документ фиксирует **достигнутые договорённости** по доработке пользовательского интерфейса расширения после обсуждения (popup vs Side Panel, контекстное меню, устойчивость состояния).  
Канон по API/WebSocket по-прежнему: **`docs/WEBSOCKET.md`**, **`docs/TECHNICAL_SPECIFICATION.md`** (в т.ч. §15, §17), **`openapi.yaml`**.

---

## 1. Цели UX

1. **Не терять контекст записи** при переключении вкладок и при закрытии UI: пользователь должен иметь возможность вернуться к активной сессии и видеть последний `conversation_id`, статус и накопленный realtime-текст (в разумных пределах).
2. **Явно понимать источник аудио**: микрофон (глобально) vs **конкретная вкладка** (`tabCapture`); в UI отображать привязку (заголовок вкладки / `tabId`, режим «Mic»).
3. **Разделить «быстрый вход» и «рабочую поверхность»**: короткие действия (логин, настройки) — отдельно от длинной работы (транскрипт, Stop, выгрузки).

---

## 2. Архитектура UI (согласовано)

### 2.1. Chrome Side Panel — основная рабочая поверхность

В **Side Panel** размещается функциональность **длинной сессии**:

- старт / останов записи (управление `MediaRecorder` и WebSocket `/ws/audio/{id}` в контексте панели, не в service worker);
- отображение **runtime / partial** расшифровки (поток с `/ws/transcript/{id}`; нормализация типов сообщений согласуется с текущим сервером, см. `docs/WEBSOCKET.md`);
- выгрузка **черновика** realtime-текста (локальный файл, отдельно от канонического export API);
- выгрузка **финальной** расшифровки и **summary** через **те же канонические HTTP**, что Web UI:  
  `GET /api/conversations/{id}/export?format=md|json` (и эндпоинт(ы) summary — по мере появления в `openapi.yaml`; см. **§5.1**);
- индикаторы: соединение WS, ошибки ASR, «идёт запись / ожидание», кнопка **Reconnect / Refresh** при обрыве;
- компактная зона статуса + прокручиваемая область транскрипта (по согласованию: допускается сворачиваемый блок);
- при **одной** Side Panel на **окно** браузера — элемент управления **выбором вкладки-контекста** (вкладки-«чипы», выпадающий список или эквивалент): для каждой вкладки, с которой пользователь запускал работу расширения, отображается **своя** сессия (`conversation_id`, статус записи, превью транскрипта). Переключение в панели **не** останавливает запись на других вкладках; см. **§3**.

**Ограничение платформы:** Side Panel — **Chromium** (Chrome / Edge). Поддержку **Firefox** и прочих не-Chromium браузеров **откладываем на последний этап** разработки расширения; до этого этапа документация и приёмка ориентируются только на Chromium. Для не-Chromium позже — отдельный fallback (например полноэкранная страница UI расширения вместо Side Panel).

### 2.2. Popup действия — только «лёгкий вход»

**Popup** (клик по иконке расширения) оставляем для:

- **авторизации** (OAuth / проверка сессии); целевой поток silent→interactive и токены **только сервиса** — [AUTH_AND_IDENTITY.md](./AUTH_AND_IDENTITY.md);
- **настроек** расширения (Server URL, источник аудио, chunk/windowed и т.д., см. `browser-extension/src/settings/storage.ts`);
- переключателя **«показать / скрыть Side Panel»** (или явная кнопка «Open panel»), с учётом требования **user gesture** для `chrome.sidePanel.open()` где применимо;
- краткий статус «залогинен / нет» и ссылка «открыть панель».

Запись, длинный транскрипт и выгрузки **не** являются обязательным содержимым popup (чтобы не бороться с закрытием popup браузером).

### 2.3. Контекстное меню иконки расширения (toolbar action)

Через **`chrome.contextMenus`** с контекстом **`action`** (MV3) добавить быстрые пункты, согласованные сейчас:

- **Start recording** (создать разговор, открыть/сфокусировать Side Panel при необходимости, начать запись в соответствии с настройками);
- **Stop recording** (безопасно: только если активная сессия; иначе disabled или no-op с уведомлением);
- **Upload audio file…** (выбор файла → `POST /api/upload` с теми же query, что сейчас в расширении);
- **Open settings** (открыть popup или страницу настроек — выбрать один канонический путь при реализации);
- опционально: **Open Side Panel**.

Поведение меню должно быть **идемпотентным** и не создавать «тихих» вторых разговоров без явного намерения пользователя.

### 2.4. Видимость Side Panel и жизненный цикл записи

- **Закрытие** Side Panel **не** останавливает уже идущую запись.
- **Повторное открытие** панели при активной записи **не** останавливает и **не** перезапускает запись и **не** создаёт новый разговор автоматически: процесс записи **независим** от того, видна панель или нет.
- **Закрытие вкладки браузера**, **привязанной к активной сессии записи** (см. §3: вкладка-контекст для микрофона или вкладка **tab capture**), должно привести к **Stop** для этой сессии **и к явному уведомлению сервера об остановке записи**: корректное завершение потока (закрытие WS `/ws/audio/{id}`, при необходимости **`finalize`** по контракту §17 / `docs/WEBSOCKET.md`), чтобы сервер не оставался в состоянии «запись продолжается» без клиента. Закрытие вкладки, **не** связанной ни с одной активной записью, на запись не влияет.
- **Закрытие окна браузера**, в котором одновременно велась запись **с нескольких вкладок**, трактуется как закрытие каждой такой вкладки: для **каждой** сессии с активной записью выполняется тот же цикл **Stop + уведомление сервера** (по очереди или параллельно — в рамках допустимого MV3; итог: ни одна активная запись окна не остаётся без явного завершения на сервере).

## 3. Состояние и восстановление (согласовано)

1. **Persist в `chrome.storage.local` / `session`** (конкретный выбор при реализации), с учётом **одной панели на окно** и **нескольких вкладок**:  
   - структура **по `windowId` + `tabId`**: для каждой «активированной» вкладки — свой блок: `conversation_id`, режим источника (mic / tab), флаг «идёт запись», привязка к `tabId` для tab capture;  
   - в панели хранится **текущий выбранный** `tabId` UI (какую вкладку пользователь смотрит в списке);  
   - последние N символов / строк realtime на вкладку (лимит по размеру storage).
2. При открытии Side Panel — **восстановление** выбранной вкладки и сессий из storage + **догонка** с сервера (`GET /api/conversations/{id}`) для выбранного контекста как источник истины для финального текста/метаданных.
3. **Service worker** не держит UI и не выполняет запись; используется для: OAuth redirect, маршрутизации сообщений, регистрации contextMenus, **`tabs.onRemoved`** / **`windows.onRemoved`** (или эквивалент): при закрытии вкладки или окна инициировать **Stop** и **сообщение серверу** для каждой затронутой активной сессии записи (см. **§2.4**).

## 4. Связь с текущим кодом и дорожной картой

- Текущая реализация в `browser-extension/` (popup + recorder + WS) считается **промежуточной**; миграция на Side Panel **не меняет** контракты `/ws/audio`, `/ws/transcript`, `POST /api/conversations`, `POST /api/upload` без отдельного решения и обновления `openapi.yaml` / `WEBSOCKET.md`.
- Раздел **§15** и **Phase B** в `TECHNICAL_SPECIFICATION.md` дополняются ссылкой на этот документ как на UX-канон расширения.
- Детали **fast/final**, **`finalize`**, метаданных сегментов — по-прежнему **§17**; при появлении новых полей WS — сначала спецификация, затем код.

## 5. Дополнительные продуктовые решения (закрыто)

### 5.1. Summary в расширении (без Web UI)

- Summary пользователь получает **только в расширении**; **переадресация в Web UI не используется** как способ получить summary.
- **Команда выгрузки / получения summary всегда доступна** в UI (не блокируется ожиданием «неизвестной» готовности).
- **Ответ сервера** — источник истины: при успехе — сохранить/показать содержимое; если summary **ещё не готов** — показать явное сообщение (текст или код из ответа API), без «тихого» пустого файла.
- **Дополнительно (рекомендация по UX):** лёгкий **polling** статуса разговора (или отдельного поля готовности summary, когда появится в API) для **индикатора** «готово / в процессе» и подсказки пользователю; polling **не** заменяет доступность кнопки и **не** дублирует бизнес-логику: итог всё равно подтверждается ответом на запрос выгрузки.

Конкретный путь HTTP фиксируется в **`openapi.yaml`** при появлении эндпоинта; до этого момента в коде допускается заглушка с тем же UX-контрактом (кнопка активна → сообщение о неготовности по согласованному статусу).

### 5.2. Несколько окон браузера

- Состояние сессий **шардируется по `windowId`**: у каждого окна свой набор вкладок-контекстов (см. §3). Глобальный «один conversation на все окна» **не** требуется.

### 5.3. Firefox и не-Chromium

- См. **§2.1**: поддержка отложена на **финальный этап**; до него приёмка и документация — **Chromium-only**.

При появлении существенных отклонений от этого документа в коде — обновить **`BROWSER_EXTENSION_UI.md`** и при необходимости добавить ADR в `docs/adr/`.
