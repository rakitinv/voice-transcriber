## 2. Цели продукта и границы

- Клиент–серверная система транскрибации: **ASR** (распознавание речи) и **LLM** (обобщения/протоколы) разделены; распознавание не выполняется «через LLM».
- Клиенты первой очереди: **Web UI**, **расширение Chromium**, **CLI**; в перспективе — другие клиенты при том же API.
- Режимы: **пакетная** загрузка аудио, **потоковая** передача (WebSocket), **post-hoc диаризация**; автоматическое включение diarization в общем серверном pipeline после ASR задаётся **только конфигурацией развёртывания** (администратор/сервер), а не переключателем в Web UI для конечного пользователя (см. §10).
- Идентификация: **OAuth2** (Google, Yandex и др. при расширении), выдача **JWT** сервиса; расширение не хранит токены провайдера; долгая сессия без повторного OAuth — через механизм **сервиса** (refresh и т.п., см. [AUTH_AND_IDENTITY.md](./AUTH_AND_IDENTITY.md)); для CLI — **API key** (см. §11).
- Хранение артефактов: **S3-совместимое** хранилище; метаданные — **PostgreSQL**; очереди — **Redis** + **Celery**.

**Пакетная загрузка файла (`POST /api/upload`):** один и тот же контракт используют **Web UI** (загрузка файла с диска или запись с микрофона до остановки → тот же multipart `file` на странице списка разговоров), **CLI** (команда `upload`, когда будет реализована в `cli/`), и отладочный скрипт `scripts/phase_a_upload_smoke.py`. **Замечание по нумерации:** WebSocket в FastAPI могут монтироваться с тем же или иным префиксом; в OpenAPI зафиксировать **точные URL**, согласованные с клиентами (в т.ч. замена `http`→`ws` для того же хоста).

### 3.4. Аутентификация, идентичность и сессия (канон)

- **Единый backend** для OAuth расширения и Web UI; секреты провайдера только на сервере.
- **Провайдер** — только для **идентичности**; хранение **`refresh_token` провайдера** для текущих сценариев **не требуется**.
- **Расширение:** гибридный UX — сначала **неинтерактивная** попытка authorize (`launchWebAuthFlow` с `interactive: false` + параметры провайдера «без UI при наличии сессии»), затем при неудаче — **интерактивный** вход; после успеха — **долгая сессия сервиса** (короткий JWT + refresh/аналог сервиса), чтобы редко показывать провайдера.
- **Web UI:** слияние нескольких провайдерских входов в одного пользователя продукта — отдельные потоки UI и данные связей «пользователь ↔ провайдерский субъект».
- HTTP-схемы запросов/ответов для расширения и refresh: **`openapi.yaml`** (версия API в `info.version`); поведение и угрозы — **[AUTH_AND_IDENTITY.md](./AUTH_AND_IDENTITY.md)**.

## 4. Решения по realtime (гибрид)

### 4.1. Принцип

- **Гибрид:** сервер задаёт **дефолт** и **жёсткие границы**; клиент (пользователь или админ deployment) выбирает режим **только внутри** разрешённого.
- Исключается рассинхрон вида «клиент запросил окно 30 с, сервер допускает максимум 10 с».

### 4.2. Режимы

- **Mode 1 — chunk:** порция аудио → ASR → частичный транскрипт.
- **Mode 2 — windowed:** скользящее/буферное окно PCM → периодический ASR → транскрипт; размер окна ограничен сервером (`max_window_ms` и др.).

### 4.3. Поставка в API лимитов

Эндпоинт **`GET /api/settings/limits`** (см. §5) возвращает в числе прочего:

- `allowed_realtime_modes` (например `["chunk","windowed"]`),
- `default_realtime_mode`,
- `max_window_ms`, границы `chunk_ms` (min/max),
- а также лимиты длительности, размера файла, TTL (для клампа на клиентах).

### 4.4. Валидация

- При **создании разговора** и при **приёме потока** сервер **повторно** валидирует выбранные параметры относительно лимитов (источник правды — сервер, не локальные настройки клиента).

### 4.5. Связь с §17

- Расширенная постановка по **сегментам по времени**, **fast/final**, **finalize**, чанкированию длинных загрузок и приоритетам очередей: **§17**; доработка realtime v2 — [REALTIME_FAST_FINAL_V2.md](./REALTIME_FAST_FINAL_V2.md).

## 5. Эндпоинты settings и контракт лимитов (`GET /api/settings/limits`)

### 5.1. `GET /api/settings/limits`

Ответ должен включать (минимум; точные имена полей — в `openapi.yaml`):

| Поле (логически) | Назначение |
|------------------|------------|
| `max_duration_seconds` | Макс. длительность одного разговора |
| `max_file_size_bytes` | Макс. размер загрузки |
| `max_ttl_days` | Макс. срок хранения |
| `allowed_realtime_modes` | Допустимые режимы realtime |
| `default_realtime_mode` | Значение по умолчанию |
| `chunk_ms_min`, `chunk_ms_max` | Диапазон размера чанка (согласованный с 500 ms–2 s и политикой сервера) |
| `max_window_ms` | Верхняя граница окна для windowed |
| `autoprolong_tail_seconds` или `autoprolong_tail_ms` | Размер «хвоста», догружаемого в разговор **A** перед стартом **B** при автопродлении (см. §7) |
| `diarization_turn_level_retranscription_default` (опционально в схеме) | Серверный дефолт: включён ли повторный ASR на каждый turn при диаризации (см. §10) |

Значения должны быть согласованы с конфигурацией сервера (например `configs/limits.yaml`), но **клиенты читают только API**.

### 5.2. `GET /api/settings/user` / `PATCH /api/settings/user`

- Хранение per-user настроек: язык по умолчанию, предпочитаемый TTL (≤ серверного), режим поиска по умолчанию (fulltext/semantic), параметры VAD для ASR (кастом по желанию), **опциональный override режима diarization** (`diarization_turn_level_retranscription_use_custom` / `diarization_turn_level_retranscription` — повторный ASR по turn **или** только расстановка спикеров; дефолт берётся из §10 / limits).
- Схема в OpenAPI; сервер при сохранении клампит значения по `limits`.

### 5.3. Клиенты

- Web UI, расширение, CLI при старте (или при логине) **запрашивают limits**, **кэшируют** и **ограничивают** локальные ползунки/поля. Настройки **diarization turn-level re-ASR** в интерфейсе задаются в **Web UI (Settings)**; расширение браузера их не дублирует.

## 7. Автопродление («бесконечная» запись): модель данных

### 7.1. Поведение

- При достижении лимита (см. ниже) разговор **A** **закрывается**, создаётся разговор **B**, запись продолжается; в UI допускается отображение как **непрерывного сеанса**; в хранилище — **цепочка** разговоров.

### 7.2. Триггер ротации

- Срабатывает **тот лимит, который наступит раньше:** превышена **длительность** записи **или** превышен **накопленный размер** аудио/артефактов (политика уточняется в OpenAPI/лимитах, но принцип «кто раньше» зафиксирован).

### 7.3. Поля данных (логическая модель)

| Поле | Описание |
|------|----------|
| `conversation_id` | UUID разговора (как сейчас) |
| `recording_session_id` | UUID **сессии записи**: для **первой** записи в цепочке **равен** `conversation_id` первого разговора; для всех **продолжений** — тот же идентификатор, что у первой записи |
| `previous_conversation_id` | Ссылка на предыдущий разговор в цепочке; у первой записи `null` |

### 7.4. Транскрипты

- **Каждый** разговор имеет **свой** транскрипт; **склейка** в один общий транскрипт на **текущем этапе** **не рассматривается** (отображение списком/цепочкой — по `recording_session_id` / `previous_conversation_id`).

### 7.5. Сбой при переключении

- Перед стартом **B** выполняется **догрузка хвоста** аудио в **A**; размер хвоста задаётся в настройках сервера и отдаётся клиенту через **`GET /api/settings/limits`** (см. §5.1).

### 7.6. LLM / summary по цепочке

- **Решение:** один **rolling** Markdown-summary на всю сессию **`recording_session_id`** (не отдельный файл summary на каждый сегмент-разговор в цепочке).
- Пересчёт summary ставится в очередь **`llm`** после успешного **финального** активного транскрипта каждого сегмента: при выключенной серверной диаризации — сразу после ASR; при включённой — после успешной **`asr_diarized`** (чтобы сводка опиралась на лучшую доступную стадию).
- Хранение: таблица **`recording_session_summaries`**, дублирование артефакта в S3 (`users/…/recording_sessions/{recording_session_id}/summary.md`). HTTP: **`GET /api/conversations/{id}/session-summary`**, плюс подсказки статуса в **`GET /api/conversations/{id}`** и флаг **`llm_session_summary_enabled`** в **`GET /api/settings/limits`**. Конфиг: **`configs/llm.yaml`** (`session_summary_enabled`, `session_summary_max_input_chars`), провайдер по умолчанию — **`ollama`** при `enabled: true`.

## 8. Фазы поставки и приёмки

Цель фаз — **инкрементальные** релизы без блокировки всего объёмом `main_prompt`.

### Phase A — базовый продукт

- OAuth (Google/Yandex) + JWT; при необходимости для разработки — ограниченный mock с изоляцией от prod.
- `POST /api/conversations`, список/просмотр/удаление разговоров.
- `POST /api/upload`, проверка размера файла.
- **`GET /api/settings/limits`**, **`GET/PATCH /api/settings/user`**.
- **`GET /api/conversations/{id}/export?format=md|json`**.
- **`GET /api/search`** (fulltext в приоритете; semantic — по готовности модели embeddings).
- Один **рабочий** путь ASR: допускается **временный stub** с фиктивными сегментами для проверки S3/БД/UI; контракт ответа **не меняется** при замене на реальную модель.
- Логирование запросов/ошибок.

**Критерий приёмки A:** пользователь может залогиниться, создать разговор, загрузить файл, увидеть статус/результат в UI, скачать export, увидеть лимиты из API.

### Phase B — realtime и расширение

- **Аутентификация расширения и сессия сервиса:** канон — [`docs/AUTH_AND_IDENTITY.md`](./AUTH_AND_IDENTITY.md); реализация OAuth/refresh пересекается с **Phase C, C7.x** в [ROADMAP.md](./ROADMAP.md).
- **UX расширения (Side Panel / popup / context menus, persist сессии):** канон — [`docs/BROWSER_EXTENSION_UI.md`](./BROWSER_EXTENSION_UI.md).
- Создание разговора **до** открытия WebSocket; расширение: `tabCapture` в manifest (для Telemost / вкладки), отправка WebM+Opus чанков на `/ws/audio/{id}`.
- Подключение клиента к **`/ws/transcript/{id}`**, договорённость о partial updates.
- Учёт лимитов длительности/размера в потоке; подготовка к автопродлению (можно в конце Phase B или начале C).
- Документирование ограничений **MV3 service worker** (при необходимости offscreen document / долгоживущий контекст).
- План замены передачи JWT **с query** WebSocket на **краткоживущий токен** или первое **текстовое** сообщение после connect (prod).

**Критерий приёмки B:** из расширения — запись с микрофона или вкладки (v1 — один источник), живой транскрипт в UI расширения, корректное завершение по лимиту.

### Phase C — расширенные возможности

- Post-hoc диаризация (WhisperX / Pyannote / гибрид по конфигу).
- Semantic search (если не вошёл в A).
- **Автопродление** по §7.
- **Dual capture** (микрофон + вкладка): **Phase 2** — микширование в **один** поток `audio.webm`, общие таймкоды относительно старта записи; в **v1** — только микрофон **или** только вкладка.
- **Метрики** (Prometheus `/metrics` на api/worker или эквивалент): время upload→202, длительность ASR, ошибки по типам, глубина очереди Celery, latency «чанк → первый partial».
- Ужесточение CORS, секретов, прод-режима OAuth.

**Критерий приёмки C:** согласованный поднабор из списка выше + регрессия A/B.

## 9. ASR pipeline (поэтапная фиксация)

Каноническая постановка по **сегментам по времени**, **fast/final**, **finalize** и гибриду доставки: **§17**. Ниже — поэтапная фиксация реализации.

Реализация приближается к `main_prompt_asr_pipeline.txt` в порядке:

1. Декод WebM/Opus → PCM (например ffmpeg/av в воркере) + тест на чанке из расширения.
2. Режим chunk: порог по времени/байтам, вызов ASR, стабильный JSON сегментов.
3. Режим window: буфер PCM, периодический ASR, политика partial vs final.
4. Merge сегментов по `start`, склейка перекрытий, спикеры до diarization.
5. Длинное аудио: нарезка по времени (например 5–10 мин), параллельные задачи Celery, merge с учётом offset.
6. Общие утилиты: bytes→PCM float32, нормализация sample rate.

**Единый механизм в коде:** один **registry/factory**, конфиг администратора выбирает провайдер и ранжирование fallback; новый провайдер = код + регистрация.

### 9.1. Плагины и fallback

- Смена движка **на лету** для **одного** разговора **не допускается**; применяется к **новым** задачам.
- Для нескольких активных провайдеров: **ранжирование**; первый ранг — умолчание; остальные активные — **fallback при ошибке** primary (политика retry/limit в OpenAPI или внутренней спецификации воркера).

## 10. Диаризация

- **Основной режим:** post-hoc после завершения записи/загрузки.
- **Включение в общий pipeline:** автоматическая постановка задачи diarization после успешного ASR включается **исключительно серверными настройками** (например `configs/diarization.yaml`: `enabled` и параметры провайдера, переменные окружения/образ воркера). В Web UI **нет** опции «всегда с диаризацией / без диаризации» для конечного пользователя, интерпретируемой как изменение этого pipeline.
- **Поведение задачи diarization (текст vs только спикеры):** в `configs/diarization.yaml` задаётся **`turn_level_retranscription`** (по умолчанию в репозитории — отключён повторный ASR по turn, остаётся расстановка спикеров по готовому ASR). Переопределение при деплое: **`VT_DIARIZATION_TURN_LEVEL_RETRANSCRIPTION`**. Пользователь может задать своё значение в **Web UI → Settings** (поля в `user.preferences`); **расширение браузера** эту настройку **не повторяет**.
- **Ручной повтор diarization:** `POST /api/conversations/{id}/diarize` (в UI — явное действие с подтверждением); не заменяет глобальное включение/отключение этапа в pipeline.
- **Повторный batch ASR** по уже загруженному аудио: **`POST /api/conversations/{id}/retranscribe`** (Web UI: **Transcribe again** на странице разговора); создаётся новая ревизия **`kind=asr`**, далее при включённой diarization возможна обычная постановка diarization.
- **Развёртывание diarization-воркера:** в `docker/docker-compose.yml` заданы **два** сервиса — `diarization-worker` (CPU wheels, build-arg `DIARIZATION_TORCH=cpu`) и `diarization-worker-gpu` (CUDA wheels, профиль compose `gpu`, `DIARIZATION_TORCH=cuda`); выбор стека PyTorch — **build-time**, не переключение в `configs/diarization.yaml` (там задаётся `device` для уже собранного образа). Команды, профили и предупреждение про одну очередь `diarization`: [docker/README.md](../docker/README.md#diarization-cpu-vs-cuda-images).
- **Детали реализации (канон):** см. [DIARIZATION_ALIGNMENT_VERSIONING.md](./DIARIZATION_ALIGNMENT_VERSIONING.md) (плагинность, отключение, merge, версии транскриптов, endpoints rerun / retranscribe, UX подтверждения).

## 17. Сегментация по времени, fast/final, доставка и метаданные транскриптов

Настоящий раздел фиксирует **согласованные решения** обсуждения (2026-04): единая ось «сегмент по времени», гибрид доставки, два параллельных результата для realtime (**fast** и **final**), явная **finalize**, схема **`kind` / `meta`**, UX Web UI и расширения, паритет экспорта. Реализация — инкрементально (Phase B+ / C); контракты WebSocket и REST при появлении новых полей — **сначала в `openapi.yaml`**, затем код.

### 17.1. Общая ось «сегмент по времени»

- **Сегмент** — логическое окно **T₀…T₁** (после нормализации к каноническому формату на вход ASR, согласованному с §9: например **16 kHz mono**).
- **Пакетная загрузка (длинный файл):** файл **делится** на сегменты по времени (с **overlap**); сегменты обрабатываются (в т.ч. **параллельно**, где допустимо), затем **merge** в единый результат **final** для разговора.
- **Realtime:** наоборот — **накопление** аудио до срабатывания политики сегмента (порог по времени / размеру окна из §4–§5) **и** до явного завершения записи (см. §17.5).
- **Два подхода к чанкам (качество vs сложность):**
  - для **загружаемых файлов** на первом этапе — **вариант 1:** независимые сегменты + merge по таймкодам/overlap; **вариант 2** (передача контекста между сегментами) — опциональное улучшение, включаемое конфигурацией;
  - для **realtime fast** — ориентир на **вариант 2** (контекст между последовательными окнами — параметры ASR / хвост текста), без смешения с обязательной параллелью «одного файла».

### 17.2. Режим доставки данных (гибрид **C**)

| Канал | Назначение | Клиенты |
|-------|------------|---------|
| **HTTP multipart** (`POST /api/upload`, см. §3.3) | Загрузка цельного файла | Web UI, расширение, CLI |
| **WebSocket бинарные кадры** (`/ws/audio/{id}`, см. `docs/WEBSOCKET.md`) | Потоковая передача realtime | Web UI (при записи с микрофона), расширение |
| **Resumable upload** (tus и т.п.) | Длинные/нестабильные сети | **Не цель v1**; бэклог |

### 17.3. Поле `kind` и JSONB `meta` (версии транскриптов)

- **`Transcript.kind`** описывает только **стадию пайплайна**, без уровня качества. Допустимые значения включают как минимум: **`asr`**, **`asr_diarized`**; новые стадии (например summary) добавляются отдельным решением.
- **`Transcript.meta`** — каноническое место для:
  - **`processing_tier`:** `fast` | `final` (уровень качества / режима обработки);
  - **`source`:** `upload` | `realtime`;
  - **`parent_revision`** (опционально) — связь с предыдущей версией при цепочках пересчёта;
  - **`finalize_id`** (опционально, строка/UUID) — **идемпотентность** операции финализации: повтор той же команды с тем же ключом не создаёт дубликат финальной задачи/строки.
- **Два параллельных результата для realtime** — это **две записи** `transcripts` (разные `revision` и/или разные строки жизненного цикла), различимые по **`meta.processing_tier`**, при одном **`kind`** (например обе `asr`), пока не запущена диаризация.
- **Диаризация (§10):** выполняется **после формирования полного разговора** (цельное аудио в хранилище и завершённая цепочка сессии по правилам продукта). Практически — **`asr_diarized`** для ветки **`final`**; ветка **`fast`** диаризации **не предполагается** в v1, если отдельно не оговорено.

### 17.4. Realtime: параллельно **fast** и **final**

- **Fast:** облегчённый режим для **минимальной задержки** отображения (тот же поток сегментов, профиль ASR/параметров «быстрый»).
- **Final:** обработка **с качеством как у пакетной загрузки** (склейка накопленного аудио сессии → тот же конвейер, что и для upload, включая нарезку/merge по §17.1).
- Это **два разных результата** (две разные обработки одних и тех же данных по смыслу пользователя), а не один результат, «тихо» заменённый вторым без следа в БД.

### 17.5. Накопление сегментов и триггеры (realtime)

- Сегмент для обработки формируется при:
  - достижении **порога** (время / окно из §4–§5), **или**
  - **явном окончании записи** пользователем.
- Обязательна **явная команда `finalize`** в протоколе WebSocket (текстовое JSON-сообщение после согласования в OpenAPI и `WEBSOCKET.md`). Она:
  - запускает построение **полного** аудио для **final** (в связке с буфером/цепочкой §7 при автопродлении);
  - вместе с **`finalize_id`** даёт **идемпотентность** при повторе после обрыва связи (клиент может **повторить** ту же команду; сервер не дублирует работу).
- **Автопродление (§7):** команда, эквивалентная **`finalize`** для текущего звена цепочки, входит в согласованный сценарий ротации **A → B** (детали сообщений — в OpenAPI / `WEBSOCKET.md` при реализации).

### 17.6. Приоритизация обработки (настраиваемая)

- **По умолчанию:** задачи **`fast`** обрабатываются **в первую очередь**; **`final`** и тяжёлые пакетные задачи — **по мере поступления** и доступности воркеров.
- Политика должна быть **конфигурируема** (очереди Celery, приоритеты, лимиты in-flight на пользователя/GPU); точные ключи — в `configs` / `docker-compose` по мере реализации.

### 17.7. Очереди Celery, GPU и брокер (рекомендации для развёртывания)

- **Разделение очередей (логически):** например `asr_fast` (низкая задержка) и `asr_final` / общая `asr` для тяжёлого offline-качества; **`diarization`** — без изменения смысла §10.
- **GPU:** развести нагрузку **API** (синхронный realtime **fast**) и **воркеры** (**final**, чанки upload, diarization), чтобы пакетный поток не «глотал» интерактив; допускается несколько сервисов воркеров с разными `VT_ASR_DEVICE` / лимитами concurrency.
- **Redis + `task_acks_late=True`:** задать **`broker_transport_options`** / **`visibility_timeout`** **больше** максимально ожидаемой длительности длинной задачи `transcribe`/merge, чтобы исключить **повторную доставку** одной и той же Celery-задачи и лишние `revision` в `transcripts`.

### 17.8. Web UI

- При наличии обеих веток — **переключатель просмотра `fast` / `final`**.
- В каждом режиме отображать **текущую стадию** актуальной версии (например **ASR** vs **ASR + диаризация**), опираясь на **`kind`** и `status` выбранной строки `transcripts`. **История всех ревизий** в UI в v1 **не обязательна** (как сейчас — достаточно актуального состояния + переключатель fast/final).
- **Повторное распознавание** (rerun от пользователя) — только в контексте **final**-ветки (не пересчитывать «быстрый черновик» как основной продуктовый результат).

### 17.9. Расширение браузера

- В отдельном поле — **текущая realtime-расшифровка (`fast`)**.
- **Кнопка выгрузки финальной** расшифровки — **неактивна**, пока **`final`** не в статусе успеха (или эквивалентная готовность в API).
- Опционально — отдельная кнопка **«сохранить текущий fast-текст»** (локально или через API — уточняется при реализации; не смешивать с каноническим **`GET /api/conversations/{id}/export`** для **final**).
- **Выгрузка финальной** расшифровки из расширения и из Web UI — **один и тот же** канонический контракт: **`GET /api/conversations/{id}/export?format=md|json`** (§3.3, §12). Выбор ветки **fast/final/auto:** query-параметр **`tier`** (см. **`openapi.yaml`** и [ROADMAP.md](./ROADMAP.md) §17); для канонического экспорта **final** клиенты используют **`tier=final`** (или **`auto`**, когда активный указатель уже на финальной версии).

### 17.10. Параметры окон и overlap

- Длины окон, overlap, лимиты чанков для **upload** и для **realtime** — **настраиваемые** (`configs/limits.yaml`, `configs/asr.yaml`, переменные окружения).
- **Начальные значения** подбираются по опыту разбора **коротких** и **длинных** файлов и по метрикам realtime; конкретные числа в ТЗ не фиксируются жёстко — задаются в конфиге репозитория и при необходимости дублируются в `docs/ROADMAP.md` как результаты калибровки.

### 17.11. Критерии готовности реализации (чеклист)

Сводка по реализации совпадает с дорожной картой: [ROADMAP.md § «ТЗ §17»](./ROADMAP.md).

- [x] **`finalize` / `finalize_id`:** канон сообщений WebSocket — [WEBSOCKET.md](./WEBSOCKET.md); в **`openapi.yaml`** для **`/ws/*`** — описательные заглушки (OAS 3 не задаёт тело WS целиком). **Автопродление (§7):** эквивалент **`finalize`** при ротации цепочки в коде **не** реализован — backlog ([ROADMAP.md](./ROADMAP.md) **C3**).
- [x] **Ветки fast/final и метаданные:** **`processing_tier`**, **`source`**, **`finalize_id`** и др. хранятся в **`Transcript.meta`** и используются пайплайном; клиент выбирает ветку через **`GET /api/conversations/{id}?tier=…`** и **`GET …/export?…&tier=…`** (описание query **`tier`** — в **`openapi.yaml`** в корне репозитория). Детальный JSON ответа содержит **`transcript_kind`**, **`transcript_status`**, ревизию и т.д.; полный объект **`meta`** по версиям при необходимости расширять (например **`GET …/transcripts`**).
- [x] **Очереди Celery:** логическое разделение **`asr_fast`** (слайсы параллельного ASR) и **`asr_final`** (полный файл, merge после chord) — см. **`workers/celery_app.py`**, **`docker-compose.yml`**. Интерактивный **fast** realtime обрабатывается на стороне API (задержка), тяжёлый **final** — воркерами.
- [x] **Длинный upload:** нарезка по времени, опционально параллель (**`VT_ASR_PARALLEL_CHUNKS`**), merge → **final** — **`workers/tasks/asr.py`**.
- [x] **Realtime:** **fast** (partial в hub) и постановка **final** после **`finalize`**; диаризация после успешного **final** ASR при включённом конфиге — как в §10.
- [x] **Web UI и расширение:** Web UI — переключатель **fast/final** и стадия на странице разговора; расширение — UX §17.9 (**«Save live text»** локально для live; **`GET /export`** с **`tier=final`** для канона). Опциональный codegen клиента только для WS — backlog ([ROADMAP.md](./ROADMAP.md)).
- [x] **`visibility_timeout`** брокера Celery (**`VT_CELERY_VISIBILITY_TIMEOUT`**, по умолчанию в compose **14400** с).
