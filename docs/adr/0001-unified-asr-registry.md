# ADR 0001: Единый механизм загрузки ASR-провайдеров

**Статус:** принято  
**Дата:** 2026-04-18

## Контекст

В кодовой базе сосуществуют:

- `plugins.loader.PluginRegistry` — задуманная динамическая загрузка из конфигурации;
- `app.asr.factory` — фабрика, мапящая имя движка на классы `WhisperASRProvider` / и т.д.

Воркеры и WebSocket опираются на `plugin_registry`, тогда как фабрика не подключена к реестру, что ведёт к дублированию и риску рассинхрона.

## Решение

1. **Единая точка создания ASR-провайдера** для приложения: один модуль (registry **или** factory после слияния), который:
   - читает активный движок и порядок fallback из конфигурации администратора (`configs/asr.yaml` + будущие поля ранжирования);
   - создаёт экземпляр `ASRProvider` для задач Celery и realtime.

2. **Переключение движка** — только конфиг/deployment; **новый тип провайдера** — изменение кода (класс + регистрация в таблице `engine → class`).

3. **Смена движка «на лету» для одного разговора** не поддерживается; новые задачи используют актуальный конфиг.

## Последствия

- Реализация Phase A/B должна **удалить или объединить** заглушки в `plugins/loader.py` с `app/asr/factory.py`.
- Документ [TECHNICAL_SPECIFICATION.md](../TECHNICAL_SPECIFICATION.md) §9 остаётся нормативным; детали ранжирования fallback — в конфиге и OpenAPI при появлении админ-API.
