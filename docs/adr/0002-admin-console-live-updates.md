# ADR 0002: почти live-обновление Ops-консоли (спринт 7)

## Статус

Принято (2026-05-14).

## Контекст

Требование [ADMIN_OPS_CONSOLE.md](../ADMIN_OPS_CONSOLE.md) §4.2: потоковое или почти-потоковое обновление (SSE / WebSocket / поллинг с backoff). Нужна совместимость с reverse proxy (таймауты, буферизация SSE), ограничение нагрузки на БД/Redis (§8) и отсутствие утечки содержимого чужих разговоров в «сигналах» обновления (§9).

## Решение

**Этап 1 (текущая реализация):** улучшенный **HTTP-поллинг** в admin-webui:

- единый хук для вкладок «Разговоры», «Инфраструктура», «Аудит», «Пайплайн», «События пайплайна»;
- **jitter** к базовому интервалу, чтобы снизить синхронные пики;
- **экспоненциальный backoff** при ошибках сети/HTTP, сброс после успешного ответа;
- **пауза** при `document.visibilityState === "hidden"` (редкий «пробуждающий» таймер вместо полного поллинга данных);
- при возврате на вкладку браузера — одно обновление и сброс интервала;
- ручное «Обновить сейчас» сбрасывает backoff (через общий `resetNonce`).

**Admin API:** добавлен лёгкий `GET /admin/api/v1/live-tick` — только `tick_ms` и `schema_version` (никаких идентификаторов разговоров и текстов). Клиент в этом спринте **не обязан** вызывать его на каждом цикле (основной поллинг идёт на существующие read-эндпоинты); эндпоинт нужен для контракта «сигнал без §9», тестов и будущей интеграции (например, сравнение версии снапшота).

## Отклонённые / отложенные варианты

- **SSE:** проще одностороннего push, но типичны проблемы буферизации в nginx без `X-Accel-Buffering: no` и настройки таймаутов; отложено до явной потребности и согласования с DevOps.
- **WebSocket:** избыточно для односторонних «тиков»; нужны sticky sessions при масштабировании — отложено.

## Откат

Вернуть прежний фиксированный поллинг: убрать jitter/visibility и оставить один `setTimeout` с ростом интервала — или отключить автообновление в UI, оставив только ручное обновление.

## Последствия

- Нет дополнительных долгоживущих соединений на админа (лимиты SSE/WS не критичны).
- Нагрузка остаётся в рамках периодических GET к уже существующим read-маршрутам; при ошибках интервал растёт, во вкладке в фоне — реже.
